# Options Flow Analytics Platform

A full-stack analytics dashboard for high-frequency options Market-By-Order (MBO) data. Ingests ES futures and SPX cash options data, stores it in Parquet format, and renders professional visualizations for quantitative analysis, flow detection, and greeks monitoring.

## Quick Start (Local Dev)

### Prerequisites
- Python 3.10+
- Node.js 18+

### 1. Install dependencies

```bash
# Python (pipeline + API)
pip install polars pyarrow fastapi "uvicorn[standard]" duckdb pytz

# Frontend
cd frontend && npm install
```

### 2. Ingest data

Drop CSV files into `data/raw/` then run:

```bash
python pipeline/ingest.py --input "data/raw/*.csv" --output "data/flow.parquet"
```

The script deduplicates on `(timestamp, Side, future_strike)`. Re-running is safe.

```bash
# Run schema unit tests
cd pipeline && python -m pytest tests/ -v
```

### 3. Start the API

```bash
# From options-flow-platform/
PARQUET_PATH=data/flow.parquet python -m uvicorn api.main:app --host 0.0.0.0 --port 8000 --reload
```

API docs: `http://localhost:8000/docs`

### 4. Start the frontend

```bash
cd frontend
node node_modules/vite/bin/vite.js --host   # dev server on http://localhost:4000
node node_modules/vite/bin/vite.js build    # production build → dist/
```

The Vite dev server proxies `/api/*` → `http://localhost:8000` automatically.

## GitHub Pages Deployment

The frontend deploys as a fully static site — no API server needed at runtime. The CI pipeline pre-generates all chart data as JSON files from the Parquet source.

### How it works

1. GitHub Actions runs `pipeline/export_static.py` which queries `data/flow.parquet` with DuckDB and writes ~20 JSON files to `frontend/public/data/`
2. Vite builds the React app; the frontend detects it's running on GitHub Pages and loads JSON directly instead of calling the API
3. The built `dist/` folder is deployed to GitHub Pages

### Setup

1. Push this repo to GitHub.

2. Enable GitHub Pages:
   - Go to **Settings → Pages**
   - Source: **GitHub Actions**

3. Push to `main` — the workflow in `.github/workflows/deploy.yml` builds and deploys automatically.

The live site will be at `https://<username>.github.io/<repo-name>/`

## API Reference

All endpoints accept optional query parameters for filtering:

| Parameter | Type | Description |
|-----------|------|-------------|
| `start` | datetime string | Filter rows after this timestamp |
| `end` | datetime string | Filter rows before this timestamp |
| `side` | `Bid` or `Ask` | Filter by order book side |
| `strike_min` / `strike_max` | float | SPX strike range |
| `dte_min` / `dte_max` | float | Time to expiry range (in years) |
| `bucket_minutes` | int | Time bucket size (1, 5, 15, 30, 60) |

### Endpoints

```
GET /signals/kpi               Dashboard KPI summary
GET /signals/basis             ES vs SPX basis per time bucket
GET /signals/pulling_stacking  P/S events overlaid on price
GET /signals/net_flow          Delta-adjusted net flow per bucket

GET /volume/heatmap            Time bucket × strike × MBO sum
GET /volume/histogram          Exploded MBO order sizes
GET /volume/spread             Bid-ask spread vs SPX price
GET /volume/depth              Order book depth by strike and side

GET /greeks/surface            3D surface (strike × DTE × greek)
GET /greeks/vanna_charm        Vanna or charm exposure by strike
GET /greeks/theta              Theta decay curves by strike
GET /greeks/scatter            Vomma vs Vega scatter
GET /greeks/correlation        Pearson correlation matrix (all 16 greeks)
```

## Domain Glossary

| Term | Description |
|------|-------------|
| **MBO** | Market-By-Order — individual order sizes at each price level, more granular than MBP |
| **Pulling** | Rapid order cancellation; clusters often precede a price drop |
| **Stacking** | Rapid order addition; clusters often precede a rally |
| **ES-SPX Basis** | ES futures typically lead SPX cash by 2–5 seconds; divergence signals an imminent move |
| **Gamma** | dDelta/dSpot — short-gamma dealers hedge dynamically, amplifying directional moves |
| **Vanna** | dDelta/dVol — IV moves cause automatic delta hedging flows |
| **Charm** | dDelta/dTime — important near expiry as time decay shifts the hedge ratio |
| **Vomma** | dVega/dVol — high vomma = nonlinear sensitivity to volatility regimes |
| **Theta** | Daily dollar cost of holding a long option; accelerates after 21 DTE |
